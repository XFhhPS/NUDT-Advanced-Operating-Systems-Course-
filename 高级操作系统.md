<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">镜像 (Image) 与 容器 (Container)：</font>

<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">原理：镜像是只读的模板，里面打包了你的应用程序及其所有的依赖库、环境变量和配置文件。容器则是镜像运行时的实例。打个比方，镜像是“图纸”，容器是根据图纸盖出来的“房子”。</font>

<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">Linux Namespace (命名空间 - 实现“隔离”)：</font>

<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">原理：这是 Linux 内核的一种机制。当一个容器启动时，Linux 会给它分配一套独立的 Namespace。在这个 Namespace 里，容器拥有自己独立的进程树、网络接口（IP）、用户 ID 等。容器以为自己独占了整个操作系统，但实际上它只是宿主机上的一个普通进程。</font>

### 一、 K8s集群的安装
本阶段的核心目标是在腾讯云公有云环境中，划定一片隔离的网络区域，并在此之上初始化 Kubernetes 控制平面与计算节点。

#### 1.1 创建集群
+ [cite_start]**实验操作：** 实验采用了腾讯云 K8s 集群服务 (TKE) [cite: 2][cite_start]。在网络配置中，创建了名为 `k8s` 的私有网络 (VPC)，分配了 `10.0.0.0/16` 的 IPv4 CIDR 网段，并划分了子网 [cite: 5, 6][cite_start]。组件配置保持了默认设置 [cite: 7]。
+ **原理解读：**
    - **TKE 托管机制：** 使用 TKE 意味着 Kubernetes 的 Master 节点（包括 API Server、etcd、Scheduler 等）由云厂商代为托管和维护。这大大降低了集群初始化的复杂性，保证了控制平面的高可用性。
    - **VPC 与 CIDR：** VPC 是公有云上逻辑隔离的网络空间。`10.0.0.0/16` 提供了一个庞大的私有 IP 地址池（约 6.5 万个可用 IP）。在 K8s 中，每一个 Pod 都会被分配一个独立的 IP 地址，庞大的底层 VPC 网段是保证集群具备充足横向扩展能力的网络基础。

#### 1.2 创建节点
+ [cite_start]**实验操作：** 在私有网络中新增了“广州六区”的子网 `10.0.2.0/24` [cite: 12][cite_start]。配置了安全组策略，一键放通了 `333`、`8001` 等后续实验需要的自定义 TCP 端口 [cite: 13][cite_start]。随后，基于上述网络配置，创建了一个包含 2 台 Ubuntu 20.04 服务器的节点池 `fanzhonghao-pool1`，并分配了免费公网 IP [cite: 14]。
+ **原理解读：**
    - **Worker 节点 (Node)：** 节点是 K8s 集群的计算工作马达。这里分配的 Ubuntu 服务器将运行 `kubelet`（与 Master 节点通信的代理）、`kube-proxy`（处理网络路由）以及容器运行时（如 Containerd）。
    - **安全组 (Security Group) 预置：** 云计算的安全模型默认拒绝所有外部入站流量。由于后续实验需要通过 NodePort 暴露服务，必须在 IaaS 层（基础设施层）提前放行对应端口，否则所有的外部 HTTP 请求都会在到达节点网卡前被云平台的虚拟防火墙丢弃。

### 二、 K8s 任务

#### SSH 登录与环境检查
+ [cite_start]**实验操作：** 通过 SSH 和公网 IP `118.126.97.226` 登录到刚创建的节点 [cite: 17][cite_start]。执行 `kubectl get pods -A` 后，观察到 `kube-system` 命名空间下运行着 `coredns`、`kube-proxy`、`tke-cni-agent` 等系统组件，而 default 命名空间为空 [cite: 18]。
+ **原理解读：** * 此时登录的是一台 Worker 节点。查看到的 `kube-system` 组件是维持集群运转的核心守护进程。例如，`coredns` 负责集群内部的服务发现（域名解析），`tke-cni-agent` 负责管理容器的网络接口，确保跨节点的 Pod 能够互相通信。

#### 2.1 通过 YAML 创建包含双容器的 Pod
+ [cite_start]**实验操作：** 编写了 `1.yaml` 文件，定义了一个名为 `last-2pod` 的 Pod，其中同时包含了基于 `nginx:1.16.1` 和 `tomcat:8.5.68` 镜像的两个容器 [cite: 20, 21][cite_start]。通过 `kubectl apply -f 1.yaml` 部署后，使用 `kubectl exec` 登入 Nginx 容器，将其主页内容修改为 "This is fanzhonghao-nginx in a pod..." [cite: 23, 24, 26][cite_start]。最后，通过 `kubectl port-forward` 将容器的 80 端口映射到本地（或绑定到 0.0.0.0 的 333 端口）并在浏览器中成功访问 [cite: 27, 28]。
+ **原理解读：**
    - **Pod 的网络命名空间共享：**  为什么要把两个应用放在同一个 Pod 里？这是因为同一个 Pod 内的所有容器共享同一个网络命名空间 (Network Namespace)。这意味着 Nginx 和 Tomcat 可以通过 `localhost` 进行极低延迟的通信。这种模式在架构设计中被称为 **Sidecar 模式**。
    - **Port-forward 的隧道机制：** `port-forward` 并不是标准的生产环境暴露服务的方式。它的原理是通过 Kubernetes API Server，在你的客户端和特定的 Pod 之间建立一条临时的 TCP 代理隧道，主要用于开发调试阶段绕过复杂的网络配置直接验证容器状态。

#### 2.2 Deployment 与 Service 的高级编排
+ [cite_start]**实验操作 (Deployment 控制器)：** 创建了一个名为 `final-dep` 的 Deployment，初始设定为 4 个 Nginx Pod 副本 [cite: 31][cite_start]。随后将其副本数缩减 (Scale) 为 1，并通过命令修改了剩余那个 Pod 的 Nginx 欢迎页内容为 "This is fanzhonghao-last-nginx in a deployment..." [cite: 33, 34][cite_start]。之后又将副本数扩容为 2 [cite: 36]。
+ **原理解读：**
    - **声明式调度与控制循环：** Deployment 控制器的核心是“维持期望状态”。当你执行缩容（4变1）或扩容（1变2）时，控制器会不断比对集群的当前状态与你的声明状态，并自动发送终止或拉起 Pod 的指令。这保障了应用在面临节点宕机等异常时具备自动恢复能力。
+ [cite_start]**实验操作 (Service 负载均衡)：** 为该 Deployment 创建了一个 Service，将 Target Port 80 暴露到 Port 8001，并最终以 NodePort 方式暴露出来 [cite: 37][cite_start]。K8s 自动分配了 `30756` 作为 NodePort [cite: 38][cite_start]。修改安全组放行该端口后，通过公网 IP 成功访问了修改后的网页 [cite: 40, 41][cite_start]。最终验证了这 2 个 Pod 分别稳定运行在 `10.0.2.7` 和 `10.0.2.4` 两个不同的节点上 [cite: 42]。
+ **原理解读：**
    - **固定入口与服务发现：**  由于 Pod 的生命周期是动态的（IP 随时可能变动），Service 对象提供了一个不变的虚拟入口。
    - **NodePort 机制：** 设置为 NodePort 后，K8s 会在集群的**每一个**工作节点上打开一个相同的端口（即 30756）。当外部 HTTP 请求到达任意一个节点的这个端口时，节点内核底层的 `kube-proxy`（通常通过 iptables 或 IPVS 规则）会充当四层负载均衡器，将流量透明地转发到后端的这两个 Nginx Pod 上。
    - **高可用调度反亲和性：** 两个 Pod 落在不同的物理节点上，体现了 K8s 调度器的容灾逻辑。如果某一节点发生硬件故障，另一节点上的 Pod 依然可以持续提供服务，实现了真正的分布式高可用。

### <font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">一、 实验设计与网络拓扑：“内连外”的精妙之处</font>
<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">实验的第一个亮点在于网络拓扑的设计。你并没有让客户端和服务器都在同一个环境中运行，而是采用了 </font>**“内连外”**<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);"> 的策略：</font>

+ <font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">[cite_start]</font>**服务器端：**<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);"> 运行在外部的宿主机 Ubuntu 14.04 上，监听 8081 端口 [cite: 480, 554]。</font>
+ <font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">[cite_start]</font>**客户端：**<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);"> 运行在内部的 QEMU 虚拟机中（IP 为 10.0.2.15），通过网关（10.0.2.2）连接外部服务器 [cite: 551, 552]。</font>

**原理解读：为什么要这么设计？**<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">  
</font><font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">[cite_start]在进行内核级 GDB 调试时，一旦触发断点，整个虚拟机的内核执行流都会被冻结 [cite: 480]。如果客户端和服务器都在 QEMU 中：</font>

1. <font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">[cite_start]</font>**终端输出混杂：**<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);"> 双方的内核栈调用会交替触发断点，导致调试信息极度混乱 [cite: 545]。</font>
2. <font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">[cite_start]</font>**网络超时假死：**<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);"> 调试服务器时，客户端的计时器仍在走，极易导致 TCP 连接握手超时失败 [cite: 545]。  
</font><font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">“内连外”完美地隔离了控制变量，使得 GDB 只需要单向冻结并追踪 QEMU 中的客户端内核栈，保证了实验的纯粹性。</font>

### <font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">13 个断点 </font>
1. **系统调用层 (**`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">sock_sendmsg</font>`**, **`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">sock_recvmsg</font>`**)：**<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);"> 用户态与内核态的边界。应用层的 </font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">send()</font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);"> 和 </font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">read()</font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);"> 就是从这里“坠入”内核的。</font>
2. **传输层 (**`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">tcp_sendmsg</font>`**, **`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">tcp_transmit_skb</font>`**)：**<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);"> TCP 协议的车间。负责将数据切片，加上 TCP 头部（端口、序列号、控制位如 SYN/ACK）。</font>
3. **网络层 (**`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">ip_queue_xmit</font>`**, **`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">ip_output</font>`**)：**<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);"> IP 协议的车间。负责加上 IP 头部（源/目的 IP），并进行路由选择和 TTL 检查。</font>
4. **邻居子系统 (**`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">dst_neigh_output</font>`**, **`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">neigh_hh_output</font>`**)：**<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);"> IP 到 MAC 的翻译官。如果不知道下一跳的 MAC 地址，这里会触发 ARP 请求。</font>
5. **设备排队层 (**`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">dev_queue_xmit</font>`**)：**<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);"> 流量控制（QoS）的关卡，准备将数据帧推入物理网卡驱动。</font>
6. **接收路径 (**`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">net_rx_action</font>`**, **`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">netif_receive_skb</font>`**, **`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">ip_rcv</font>`**)：**<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);"> 软中断入口。网卡收到电信号后，通过硬件中断通知 CPU，随后 CPU 调度软中断来从网卡拉取数据，一路向上剥离头部。</font>

### <font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">三、 核心实验节点深度剖析（时间线追踪）</font>
<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">报告通过 6 个关键节点，完整记录了 TCP 的一生。</font>

#### <font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">阶段一：建立连接（Node 1 & 2 - 三次握手）</font>
<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">[cite_start]当客户端调用 </font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">connect()</font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);"> 时，触发了复杂的底层交互 [cite: 670]：</font>

1. <font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">[cite_start]</font>**发送 SYN 报文：**<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);"> 客户端 TCP 层调用 </font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">tcp_transmit_skb</font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">，此时观察到报文长度 </font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">len=0</font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">，说明这是一个纯控制报文（只有 20 字节 TCP 头，无数据）[cite: 696]。</font>
2. <font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">[cite_start]</font>**插曲：ARP 解析：**<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);"> 当报文走到邻居子系统 </font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">dst_neigh_output</font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);"> 时，内核发现不知道网关 10.0.2.2 的 MAC 地址。于是，</font>**原本的 SYN 报文被暂停，内核先伪造了一个 ARP 请求**<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">（EtherType=0x0806）交由 </font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">dev_queue_xmit</font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);"> 发送 [cite: 698]。</font>
3. <font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">[cite_start]</font>**握手完成：**<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);"> ARP 解析完成后，SYN 顺利发出。随后软中断 </font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">net_rx_action</font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);"> 收到服务器的 SYN-ACK，IP 层 </font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">ip_rcv</font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);"> 接收后，TCP 层自动回复纯 ACK（长度再次为 0）[cite: 699, 708][cite_start]。此时，用户态的 </font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">connect()</font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);"> 终于解阻塞返回，“已连接到服务器” [cite: 671, 711]。</font>

#### <font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">阶段二：数据收发验证（Node 3, 4, 5 - 封装与拆解）</font>
<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">[cite_start]这一阶段极其精彩地展示了</font>**数据封装（Encapsulation）**<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);"> 的洋葱模型 [cite: 688]。  
</font><font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">[cite_start]客户端调用 </font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">send()</font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);"> 发送 21 字节的 “客户端测试数据” [cite: 527]：</font>

+ <font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">[cite_start]</font>**TCP 层 (**`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">tcp_transmit_skb</font>`**)：**<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);"> 记录到长度 </font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">$34 = 21</font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);"> [cite: 719][cite_start]。加上 20 字节 TCP 头，带有 </font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">PSH+ACK</font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);"> 标志 [cite: 716, 719]。</font>
+ <font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">[cite_start]</font>**IP 层 (**`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">ip_queue_xmit</font>`**)：**<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);"> 记录到长度 </font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">$35 = 41</font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);"> (21 + 20) [cite: 716][cite_start]。加上 20 字节 IP 头，最终 IP 总长 </font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">$37 = 61</font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);"> (41 + 20) [cite: 721]。</font>
+ <font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">[cite_start]</font>**链路层 (**`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">dev_queue_xmit</font>`**)：**<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);"> 记录到长度 </font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">$39 = 75</font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);"> [cite: 722][cite_start]。加上 14 字节的以太网 MAC 头部 (61 + 14 = 75 字节) [cite: 722]。以太网帧正式送入网卡。</font>

<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">随后，服务器响应数据。客户端的 RX（接收）路径被激活：</font>

+ <font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">[cite_start]底层网卡不断收到分段数据，疯狂触发 </font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">ip_rcv</font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);"> 和 </font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">net_rx_action</font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);"> 软中断 [cite: 727]。</font>
+ <font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">[cite_start]数据在 TCP 层重组完毕后，终于触发 </font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">sock_recvmsg</font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">，沉睡在 </font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">read()</font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);"> 函数上的应用层进程被唤醒，顺利打印出“服务器响应: 数据已接收” [cite: 727]。</font>

#### <font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">阶段三：断开连接（Node 6 - 四次挥手）</font>
<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">[cite_start]客户端执行 </font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">close(sock)</font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">，并不意味着瞬间切断网线，而是向内核下达了主动关闭的指令 [cite: 687, 738]。</font>

+ <font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">[cite_start]</font>**发送 FIN：**<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);"> 内核再次调用 </font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">ip_queue_xmit</font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);"> 等函数。通过调试信息 </font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">0x50 0x11</font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);"> 可以精准破译：</font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">0x50</font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);"> 是 20 字节头部，</font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">0x11</font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);"> 二进制为 </font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">00010001</font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">，完美对应了 TCP 的 </font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">FIN + ACK</font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);"> 标志位 [cite: 737]。</font>
+ <font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">[cite_start]</font>**接收挥手：**<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);"> 随后再次触发 </font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">ip_rcv</font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);"> 收到服务器的确认 [cite: 738][cite_start]。四次挥手走完，TCP 状态机进入 </font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">TIME_WAIT</font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">，连接正式在内核中销毁 [cite: 738, 739]。</font>

### <font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">四、 宏观总结</font>
<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">[cite_start]通过这份实验报告，我们可以将 Linux 复杂的网络组件归纳为两条清晰的“高速公路” [cite: 688, 690]：</font>

1. **发送路径（自顶向下，由系统调用驱动）：**<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">  
</font><font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">[cite_start]</font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">User send() -> VFS -> Socket -> TCP (切片加头) -> IP (路由查表加头) -> Neighbor (MAC寻址) -> Qdisc (排队) -> NIC Driver (DMA 发送)</font>`<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);"> [cite: 688]。</font>
2. **接收路径（自底向上，由硬件中断与软中断驱动）：**<font style="color:rgb(31, 31, 31);background-color:rgb(233, 238, 246);">
